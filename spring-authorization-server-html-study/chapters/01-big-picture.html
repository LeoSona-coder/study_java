本章解决什么问题？

你准备新增 conference-module-auth，但原来的 system 模块又已经有用户、角色、菜单、按钮权限。 那么 auth 和 system 会不会重复？Gateway 到底该干什么？业务服务还需不需要自己验 JWT？

本章先把大图画清楚。只要大图清楚，后面的配置就不会乱。

官方依据： 本章基于 Spring Authorization Server 1.5.8 Reference 的 Overview 页面。 Overview 说明 Spring Authorization Server 提供 OAuth2 / OpenID Connect 相关规范实现，并构建在 Spring Security 之上。 本章对 conference-module-auth、Gateway、system、业务服务的职责划分，属于项目落地建议，不是官方原文。

先讲人话：auth 是发证机关，不是业务服务

你可以把 conference-module-auth 想成“发证机关”： 它不负责会议报名，不负责支付，不负责文件存储，也不负责渲染菜单树。

它负责回答一个更底层的问题： 这个 client 是否被允许申请 Token？ 这个用户是否已经完成认证？ 这次授权请求是否合法？ 如果合法，应该签发什么 Token？

先记住： Authorization Server 是发 Token 的地方。 Resource Server 是验 Token 的地方。 system 是权限档案室。 Gateway 是入口门卫。

conference OAuth2 总体架构图

四个边界一定要分开

1. auth：Token 签发中心

conference-module-auth 负责 OAuth2 / OIDC 协议端点、client 校验、授权码、Token、JWK、issuer 等。

2. system：权限档案室

conference-module-system 继续管理用户、角色、菜单、按钮权限。它提供权限数据，但不应该成为 OAuth2 Token 签发器。

3. Gateway：入口门卫

Gateway 面向外部请求，校验用户 JWT。验证通过后，再向下游服务传递可信的 user_id、tenant_id、authorities 等上下文。

4. 业务服务：只做业务

conference、pay、infra 主要处理业务逻辑。它们可以做方法级权限校验，但不要各自实现一套登录和 Token 签发。

项目落地：Gateway 到底验什么？

这是项目落地建议，不是官方原文。

浏览器请求进来时，Gateway 应该把自己当成 Resource Server： 它校验 access_token 的签名、issuer、过期时间、必要 claims。 校验通过后，Gateway 才能向下游传递可信用户上下文。

这样下游服务不用重复解析外部 token，也不用信任前端直接传来的 X-User-Id。这些 header 只能由 Gateway 注入，外部传入的同名 header 应该被清理。

别在这里翻车

  • 把 system 当授权服务器：短期省事，长期会把用户权限、登录流程、OAuth2 协议端点混在一起。
  • Gateway 只转发不验签：那它就不是门卫，只是一个路由器。
  • 业务服务相信外部 header：如果没有清理外部伪造的 X-User-Id，权限边界会被绕过。
  • 用户 Token 和服务 Token 混用：用户访问和服务间调用是两条链路，第 8 章会专门拆。

这一章记住这几句话

Authorization Server 是发 Token 的地方。 Resource Server 是验 Token 的地方。 system 是权限档案室，不是 Token 签发器。 Gateway 是入口门卫，要校验 JWT，并向下游传递可信上下文。
