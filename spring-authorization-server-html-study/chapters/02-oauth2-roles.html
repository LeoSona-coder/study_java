OAuth2 角色：这些人到底是谁？

OAuth2 学不明白，通常不是因为代码难，而是因为一开始就把 user、client、resource server 混成了一团。

本章解决什么问题？

你之前问过：platform-admin、conference-portal 注册成 public PKCE client， gateway、system、infra、conference、pay 注册成 confidential service client，这是什么意思？

本章就把这些“人”对号入座。 先记住一个关键点：user 不是 client，client_id 不是 user_id。

官方依据： 本章基于 Spring Authorization Server 1.5.8 Reference 的 Overview 与 Core Model / Components 页面。 Overview 的 Feature List 包含 Authorization Code、Client Credentials、Token Formats、Client Authentication Methods，并列出 none 作为 public clients 的客户端认证方式。 Core Model / Components 说明 RegisteredClient 是注册到授权服务器的 client 表示，client 在发起授权流程前必须注册，并拥有 client_id、可选 client_secret、redirect_uri、scope、grant type 等元数据。 本章把这些概念映射到 conference 项目，是项目落地解释，不是官方原文。

先讲人话：OAuth2 不是“用户拿密码换 token”这么简单

在 OAuth2 里，系统关心的不只是“用户是谁”，还关心“哪个 client 在替用户请求资源”。

举个例子：张三登录了后台。 张三是用户，platform-admin 是 client。 授权服务器要同时判断： 张三是否登录成功？ platform-admin 是否是合法 client？ 它申请的 redirect_uri、scope、grant type 是否允许？

先记住： 用户认证解决“人是谁”。 client 认证解决“应用是谁”。 OAuth2 授权流程会同时关心这两件事。

OAuth2 四个核心角色

Resource Owner

资源拥有者。放到 conference 项目里，就是登录用户，比如平台管理员、会议主办方、参会用户。

Client

代表用户或代表服务去申请 Token 的应用。比如 platform-admin、conference-portal，或者 conference、pay、infra 这些内部服务。

Authorization Server

授权服务器。你的项目里就是 conference-module-auth，负责校验授权请求并签发 Token。

Resource Server

资源服务器。Gateway 和业务服务都可能承担这个角色：它们接收请求，并基于 Token 或可信上下文做访问控制。

映射到 conference 项目

public client 和 confidential client

在授权服务器里，client 是要注册的。 它不是某个登录用户，而是一个要向授权服务器申请 Token 的应用。

Public Client

适合浏览器端 / SPA

  • 例如：platform-admin、conference-portal
  • 不能安全保存 client_secret
  • 使用 Authorization Code + PKCE
  • 客户端认证方式通常是 none
  • 必须严格校验 redirect_uri

Confidential Client

适合后端服务

  • 例如：gateway、system、conference、pay、infra
  • 可以保存 client_secret
  • 服务间调用常用 client_credentials
  • 可以使用 client_secret_basic
  • secret 应放配置中心或密钥系统，并避免明文泄露

Demo 不等于生产：client_id 不是 user_id

你可能在 demo 里看到 client-a、messaging-client 这种名字。 它们不是用户账号，而是注册到授权服务器的客户端。

生产中也不要用用户 ID 当 client_id。 client_id 表示应用身份，比如 platform-admin 或 pay-service； user_id 表示登录用户，比如某个管理员或参会人。

项目落地：conference 应该怎么注册 client？

这是项目落地建议，不是官方原文。

platform-admin public client，Authorization Code + PKCE，用于平台管理后台。
conference-portal public client，Authorization Code + PKCE，用于会议门户端。
gateway confidential client，可用于服务身份场景；同时 Gateway 也是外部请求的资源服务器入口。
conference / pay / infra / system confidential service clients，服务间调用时通过 client_credentials 获取服务 Token。

别在这里翻车

  • 把用户当 client：用户是 Resource Owner，client 是应用。两者不是一回事。
  • 把 client_secret 放进浏览器：浏览器端无法安全保存 secret，所以 public client 不应该依赖 secret。
  • 把所有服务共用一个 client：这样审计和权限收敛会很差。pay、infra、conference 最好有自己的 client 身份。
  • 混用用户 Token 和服务 Token：用户 Token 表示“某个用户访问系统”，服务 Token 表示“某个服务访问另一个服务”。

这一章记住这几句话

user 不是 client，client_id 不是 user_id。 platform-admin、conference-portal 是 public client，不能保存 secret，应该走 Authorization Code + PKCE。 gateway、system、conference、pay、infra 是 confidential client，可以使用 client_credentials 做服务间调用。 conference-module-auth 是 Authorization Server，Gateway 和业务服务是 Resource Server。
