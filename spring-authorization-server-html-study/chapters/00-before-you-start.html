本章解决什么问题？

很多后端同学一看到 Spring Authorization Server，就马上去搜： “最小配置怎么写？”、“JWKSource 怎么配？”、“怎么发 JWT？”

但如果你先写代码，后想边界，后面大概率会出现这种局面： system 也能登录，auth 也能发 Token，Gateway 也在查权限，业务服务也在解析用户。 最后谁都能管一点，谁都说不清责任。

官方依据： 本章基于 Spring Authorization Server 1.5.8 Reference 的 Overview 与 Getting Help 页面。 Overview 说明 Spring Authorization Server 是基于 Spring Security 构建的 OAuth2 / OpenID Connect 授权服务器框架； Getting Help 页面建议阅读官方文档、How-to Guides、Samples，并通过 Stack Overflow 与 GitHub 获取帮助。 本章关于 conference 项目模块拆分的内容，是项目落地建议，不是官方原文。

先讲人话：它不是普通 login-service

普通 login-service 的思路通常是： 用户名密码进来，查用户表，生成一个 token，塞 Redis 或签 JWT，然后返回。

Spring Authorization Server 的思路不是这样。 它站在 OAuth2 / OIDC 协议视角看问题： 谁是客户端？谁是用户？谁能申请什么 scope？用什么授权方式？ token 发给谁？资源服务器怎么相信这个 token？

先记住： 登录只是授权服务器的一部分。 Authorization Server 更大的职责是：按协议校验 client、校验授权请求、签发 token、暴露协议端点。

学习地图：先大图，再配置

哪些官方页面必须认真读？

1.5.8 Reference 首页把文档分成几块：Overview、Getting Help、Getting Started、 Configuration Model、Core Model / Components、Protocol Endpoints、How-to Guides。 对你这个项目来说，阅读优先级可以这样排：

第一优先级

Overview、Getting Started、Configuration Model、Core Model / Components。

第二优先级

Protocol Endpoints、SPA + PKCE、JPA core services、Add authorities as custom claims。

后置阅读

Social Login、Extension Grant、Dynamic Client Registration、多租户等，等主链路跑通再看。

项目落地：为什么需要独立 conference-module-auth？

这是项目落地建议，不是官方原文。

你的 system 模块已经有用户、角色、菜单、按钮权限。它更像“权限档案室”： 保存谁是谁、有什么角色、能看到哪些菜单、能点哪些按钮。

但 OAuth2 / OIDC 授权服务器要处理的是另一类问题： client 注册、授权请求、授权码、Token 签发、JWK、OIDC 元数据、Token 生命周期。 这些职责如果继续塞进 system，后面会很难维护。

所以更清晰的拆法是： conference-module-auth 负责协议和 Token， conference-module-system 负责用户和权限数据。

别在这里翻车

  • 不要把 Spring Authorization Server 当成“JWT 工具类”。
  • 不要让 system、auth、gateway 都各自查一遍权限。
  • 不要把 demo 里的内存 client、随机密钥直接带到生产。
  • 不要把“用户登录成功”误认为“OAuth2 授权流程设计完成”。

这一章记住这几句话

Spring Authorization Server 是授权服务器框架，不是普通登录接口。 先定边界，再写配置。 在 conference 项目里，auth 发 Token，system 管权限档案，Gateway 守入口，业务服务信任经过 Gateway 处理后的上下文。
